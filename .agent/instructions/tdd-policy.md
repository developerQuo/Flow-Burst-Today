# TDD 정책 (Test Driven Development)

## 요약
- 기본 개발 방식은 TDD(Red → Green → Refactor)
- 테스트를 먼저 작성하고 실패를 확인한 뒤 구현
- 리팩터링은 테스트 통과 상태에서만 수행
- 문서/주석 변경은 예외 가능(명시 필요)

이 프로젝트는 켄트 벡(Kent Beck)이 주창한 TDD(Test-Driven Development) 원칙을 철저히 준수하여 개발한다. 우리의 목표는 단순하다: **"작동하는 깔끔한 코드 (Clean Code That Works)"**

## 1. TDD의 핵심 주기: Red - Green - Refactor

모든 기능 구현은 다음 세 단계를 반복하며 진행한다.

### 🔴 Red: 실패하는 테스트 작성
- **목표**: 구현할 기능의 요구사항을 테스트 코드로 정의한다.
- **원칙**:
    - 컴파일 에러도 '실패'로 간주한다.
    - 테스트는 우리가 무엇을 만들려고 하는지(What)를 명확히 한다.
    - 기존 테스트가 통과하는 상태에서, 새로운 기능을 위한 테스트를 단 하나만 작성한다.
    - 이 테스트가 실패하는 것을 눈으로 확인하기 전에는 절대 프로덕션 코드를 작성하지 않는다.

### 🟢 Green: 테스트를 통과하는 최소한의 코드 작성
- **목표**: 어떻게든(죄악을 저지르더라도) 테스트를 통과시킨다.
- **원칙**:
    - 빨리 녹색 막대(Green Bar)를 보는 것이 최우선이다.
    - 하드코딩을 하거나, 더러운 코드를 작성해도 좋다. 정답을 구현하려 하지 말고 일단 통과만 시켜라.
    - 오직 '현재 실패하는 테스트 하나'를 통과시키기 위한 코드만 작성한다.

### 🔵 Refactor: 중복 제거 및 구조 개선
- **목표**: 행동을 유지하면서 코드의 구조를 개선한다.
- **원칙**:
    - 모든 테스트가 통과하는 상태에서만 수행한다.
    - **중복을 제거**한다. (가장 중요)
    - 의도를 명확하게 드러내도록 변수명/메서드명을 변경하고 구조를 다듬는다.
    - 리팩터링 중 테스트가 깨지면 즉시 되돌리거나 수정한다.

## 2. 켄트 벡의 두 가지 절대 규칙 (The Two Rules)

1. **오직 자동화된 테스트가 실패할 때만 새로운 코드를 작성한다.**
    - 추측성 코딩(Speculative Coding) 금지. "나중에 필요할 것 같아서" 작성하는 코드는 허용하지 않는다.
2. **중복을 제거한다.**
    - 중복은 모든 악의 근원이다. 코드의 중복, 로직의 중복, 테스트와 프로덕션 코드 간의 정보 중복 등을 찾아 제거한다.

## 3. 실전 가이드라인

- **Baby Steps (아기 걸음마)**:
    - 보폭을 작게 가져간다. 너무 큰 테스트를 작성하지 말고, 아주 작은 기능 단위로 나누어 테스트한다.
    - 막히면 더 작은 단위로 쪼갠다.
- **테스트는 '행동'을 검증한다**:
    - 내부 구현 상세(Implementation Detail)가 아닌 인터페이스와 객체의 행동(Behavior)을 테스트한다.
    - 그래야 리팩터링 시 테스트가 깨지지 않고 우리를 지켜줄 수 있다.
- **테스트 격리**:
    - 각 테스트는 서로 독립적이어야 하며 순서에 의존하지 않아야 한다.
- **예외 상황 (3A Pattern)**:
    - Arrange (준비) - Act (실행) - Assert (단언) 패턴을 명확히 구분하여 가독성을 높인다.

---
*"코드가 제대로 작동하는지 불안해하며 수정하는 스트레스를 없애고, 언제든 자신 있게 코드를 변경할 수 있는 자유를 얻자."*
