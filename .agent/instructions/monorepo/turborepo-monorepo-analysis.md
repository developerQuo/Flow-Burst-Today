# Turborepo 모노레포 적용성 분석 (Flow-Burst-Today)

## 문서 목적

- 현재 저장소 상태를 기준으로 Turborepo 모노레포 적용 타당성을 판단한다.
- 프로젝트 관리, 빌드 최적화, 의존성 관리, 공용 모듈 설계의 우선순위를 정한다.

## 현재 코드베이스 진단 (사실 기반)

1. 앱 경계
- `web/`: Next.js 14 + Jest + Storybook + Chromatic까지 갖춘 주 개발 축
- `mobile/`: Expo 기본 템플릿 수준 (현재 도메인 로직/테스트 체계 미구축)
- `shared/`: `types/native-bridge.d.ts`만 존재하며 패키지화/실사용 연결 없음

2. 패키지/버전 상태
- `web/`에 `package-lock.json`과 `yarn.lock`이 함께 존재해 패키지 매니저 정책이 불명확함
- React 버전 분리: `web`(18.3.1), `mobile`(19.1.0)
- TypeScript 버전 분리: `web`(^5.5.3), `mobile`(~5.9.2)

3. 루트 운영 상태
- 루트 워크스페이스 설정(`package.json` workspaces, `turbo.json`) 부재
- `.github/workflows`는 루트에서 `npm ci`를 수행하지만 루트 패키지 기준이 없어 유지보수성이 낮음

## Turborepo 적용 시 기대 가치

1. 프로젝트 관리
- 앱별 스크립트 난립을 루트 `turbo run` 명령으로 통합할 수 있다.
- CI/CD에서 동일한 진입점을 사용해 운영 표준을 만들 수 있다.

2. 빌드 최적화
- `web`의 빌드/테스트/스토리북 캐시 적중으로 반복 실행 시간을 줄일 수 있다.
- 변경 없는 패키지를 건너뛰는 affected 실행이 가능해진다.

3. 의존성 관리
- 단일 lockfile과 workspace 프로토콜로 버전 드리프트를 제어할 수 있다.
- 공통 설정 패키지(ESLint/TSConfig)로 규칙 차이를 줄일 수 있다.

4. 공용 모듈
- 현재 `shared/types`를 실제 패키지로 승격해 웹/모바일 계약 타입을 강제할 수 있다.
- 이후 순수 도메인 로직 분리로 플랫폼 독립 코드 재사용을 늘릴 수 있다.

## 효과성 평가

1. 프로젝트 관리 효과: 높음
- 현재 앱이 분리되어 있어 워크스페이스 전환 이득이 즉시 발생한다.

2. 빌드 최적화 효과: 중간~높음
- `web`은 즉시 효과가 크고, `mobile`은 테스트/빌드 파이프라인 정비 후 효과가 커진다.

3. 의존성 관리 효과: 높음
- 락파일/버전 정책 불일치가 명확하므로 개선 여지가 크다.

4. 공용 모듈 효과: 중간
- 타입 공유는 즉시 가능하지만, 런타임 공유는 React 18/19 차이와 플랫폼 API 차이 때문에 단계적 접근이 필요하다.

## 결론 및 권장 순서

1. 워크스페이스 + Turborepo 뼈대 구성
2. 패키지 매니저 단일화 및 lockfile 일원화
3. 빌드/테스트 파이프라인 캐시 적용
4. `shared/types` 패키지화
5. 순수 도메인 로직 공용화 (UI/플랫폼 코드는 후순위)

위 순서로 진행하면 초기 리스크를 낮추면서도 운영/속도 개선을 빠르게 체감할 수 있다.
